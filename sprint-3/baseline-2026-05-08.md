## Hot-spots identifiés (à creuser au début Sprint 3)

D'après les `audits` détaillés des 10 fichiers JSON dans `/tmp/lh-s3-prod/` :

### 1. `mainthread-work-breakdown` mobile : 4-11 s
Présent sur toutes les URLs mobile. Cause probable : framer-motion globaux (header, sections home, mega-menu), bundles `@react-three/fiber` + `three.js` chargés en initial sur la home, code-splitting incomplet entre routes.

### 2. `largest-contentful-paint` mobile : 2,7 - 5,2 s
- `/blog` : 5,2 s — probablement les 3 cards images (mêmes featuredImage qui servent partout) qui chargent en parallèle au mount.
- `/agence` : 5,1 s — sans doute hero parallax + portraits équipe.
- `/services/publicite-en-ligne` : 3,7 s — heroes services.

### 3. `total-blocking-time` mobile : 430 - 850 ms
- Pire sur `/services/publicite-meta` (850 ms) et `/services/publicite-en-ligne` / `/agence` (760 ms).
- Indique du JS lourd qui bloque le main thread après FCP.

### 4. A11y 80-83 sur les pages non-blog
- A11y `/blog` desktop : 91 (le travail Sprint 2 sur le blog a payé).
- A11y `/agence` desktop : 92.
- A11y `/` mobile : 80 — point bas. À auditer avec axe-core pour identifier les violations exactes (boutons sans label, contrastes, target-size).

### 5. CLS 0,095 sur `/`
- Frôle le seuil Google 0,1. Probablement vidéos lazy ou images sans `width/height` qui shift le layout au load.

## Note méthodo : preview vs prod

Une première mesure sur le Deploy Preview Netlify (`https://deploy-preview-2--site-web-digiqo.netlify.app`) a donné des scores ~10-20 points plus bas que la prod. Cause : Netlify injecte sur ses Deploy Previews une toolbar de feedback qui charge ~1,3 MB de vidéos `app.netlify.com/cdp/video/deploy-previews-*.mp4` plus du JS supplémentaire. Cette toolbar n'existe **pas en prod** sur `digiqo.fr`. Toutes les mesures Sprint 3 doivent donc être prises sur la prod (ce baseline) ou sur un Deploy Preview désactivant la toolbar.

Snapshot des deux baselines pour traçabilité (si besoin de re-comparer) :
- Preview commit `5e7ecc81` mesuré 11:04 UTC : voir `/tmp/lh-s3-baseline/*.json`
- Prod digiqo.fr post-merge mesuré 11:10 UTC : voir `/tmp/lh-s3-prod/*.json` (ce baseline)

## Workflow Sprint 3

Branche : `seo-geo-rattrapage-s3-perf`. Approche par hot-spot ciblé :

1. Choisir un hot-spot (ex: « mainthread-work-breakdown sur `/` mobile »).
2. Hypothèse : « le bundle `@react-three/fiber` est dans le initial chunk → dynamic-importer ».
3. Implémenter en 1 commit `[S3-#N]`.
4. Re-mesurer Lighthouse PROD sur les 5 URLs.
5. Inscrire le delta dans ce fichier (table « Suivi » à ajouter à chaque commit).
6. Si gain insuffisant ou régression : rollback du commit, hypothèse suivante.

À chaque commit : delta before/after attendu **sur l'URL ciblée**, et vérification qu'il n'y a pas de régression sur les 4 autres URLs.

Cible Sprint 3 livrable : tableau before/after final dans la PR récapitulant les 10 mesures — ce baseline est la colonne « before ».
